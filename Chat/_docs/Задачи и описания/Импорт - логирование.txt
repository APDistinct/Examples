В общем. Мы хотим видеть в БД наиболее полную информацию о обновлениях. У нас есть два типа обновления: структура и сегменты. Каждое их них выполняется для конкретного консультанта (личный номер).
1. Поэтому нам надо писать в БД инфо о факте обновления:
  а. Дата старта
  б. Тип обновления (структура или сегменты)
  в. Завершение (успешно(true), неуспешно(false) или в процессе (null))
  г. Дата завершения
  д. Количество записей
  Для обновления структуры:
  е. Количество новичков
  ж. Количество обновлённых
  
  Для обновления сегментов:
  указывать, если появились новые сегменты (результат функции SyncPartnerSegments) и какие

2. О косяках обновления:
  сейчас  у нас три вида косяка структуры: дубликат телефона, дубликат мыла, не найден наставник.
  Записывать в базу эти три косяка в доп таблицу с указанием типа, изначального значения, номера и идентификатора консультанта.
  Для этого надо доработать интерфейс IListener, что бы он различал эти три вида косяка.
  
В импортировании сегментов - там две функции. Одна из них устаревшая.
правильная - public static int ImportSegments(this ChatEntities entities, IEnumerable<Tuple<int, IEnumerable<string>>> users)
где консультант передаётся со списком его сегментов. 
сейчас ХП User_UpdateSegments выполняется без получения результата выполнения, хотя в БД она возвращает количество добавленных сегментов и удалённых для консульнанта. Хорошо бы получать этот результат и суммировать эти параметры и возвращать как результат выполнения и тоже сохранять в базу.

Как делать.
Для обновления структуры сделать класс, наследующий IListener. В констуруктор передавать номер консультанта, для которого выполняется обновление, он сохраняет запись в базу. Слушает процесс выполнения, сохраняет в базу косяки и по завершению закрывает запись в БД (результат и время завершения).
В обновления сегментов проще, там по сути только начало и конец, интерфейса слушателя нет (ничего не сообщается о процессе). 

Количество записываемой в БД информации можно увелить по желанию.
